# Progress Log

## Learnings
- Prisma 7.x: datasource URL goes in `prisma.config.ts` (not `schema.prisma`). The `url` field in schema.prisma is rejected.
- `npx prisma validate` is the correct command to validate schema syntax without a DB connection.
- `npx prisma generate` works without a live DB (generates TS client from schema).
- `npx tsc --noEmit` validates TypeScript without emitting files.
- When no PostgreSQL is available locally, migration SQL is written manually and validated via schema validation + typecheck.
- Project stack: Next.js 15 (App Router), TypeScript, Prisma 7, PostgreSQL. Code lives in `app/` subdirectory.
- Prisma 7 PrismaClient constructor requires either `accelerateUrl` or `adapter` — TS type rejects plain `{ log: [...] }` options. Workaround: cast constructor to `any` and type the result as `PrismaClient`.

## Iteration 2 — US-005: Asset core schema and migration
- What was done: The `Asset` model was already fully defined in `schema.prisma` from a prior session (including all enums `AssetStatus`, `OwnershipModel`, all FK relations to AssetFamily, AssetType, LifecycleStage, ResponsibleBody x5, self-referential parent, and all nullable stub FKs for Phase 2/3). Created migration SQL at `prisma/migrations/0005_assets/migration.sql` with: two CREATE TYPE enums, CREATE TABLE assets with all columns and FK constraints (Phase 3 FK stubs omitted from REFERENCES since target tables don't exist yet), and 5 CREATE INDEX statements. Ran `prisma validate` → valid. Ran `prisma generate` → success. Ran `tsc --noEmit` → clean.
- Files affected: `app/prisma/migrations/0005_assets/migration.sql`
- Learnings for next cycles:
  - Patterns: Phase 3/future FK stubs (planning_entity_id, developer_obligation_id, active_budget_envelope_id, location_id) are declared as plain UUID columns without REFERENCES in the SQL — the FK constraints will be added via ALTER TABLE in the migrations where those target tables are created.
  - Traps: Do not add REFERENCES for tables that don't exist yet in earlier migrations; it causes FK constraint errors at runtime even if Prisma schema compiles fine.
  - Helpful context: `defaultMode` in settings.local.json belongs inside the `permissions` object per Claude Code schema validation, not at root level.

## Iteration 5 — US-008: Event schema and migration
- What was done: Added `EventCategory` enum, `EventType` model, and `Event` model to `prisma/schema.prisma`. Added `events` reverse relation to `Asset`, `LifecycleStage`, and `ResponsibleBody` (named relation `"EventResponsibleBody"` to avoid ambiguity). `recorded_by_user_id` is a plain UUID nullable stub — no REFERENCES since users table created in US-018. Created migration SQL at `prisma/migrations/0008_events/migration.sql` with: CREATE TYPE event_category, CREATE TABLE event_types, INSERT seed data for all 34 event types (10 business, 16 operational, 8 governance), CREATE TABLE events with all columns, and 4 indexes. Ran `prisma validate` → valid, `prisma generate` → success, `tsc --noEmit` → clean. Marked US-008 complete in PRD.md.
- Files affected: `app/prisma/schema.prisma`, `app/prisma/migrations/0008_events/migration.sql`
- Learnings for next cycles:
  - Patterns: When a model has multiple FK relations to the same table (e.g., ResponsibleBody), each relation needs a unique Prisma relation name to avoid ambiguity errors.
  - Patterns: Events table has no `updated_at` — it is append-only by design; only `created_at` is needed.

  - Patterns: Duplicate check for updates must exclude the record being updated: `id: { not: id }`.

## Iteration 8 — US-011: Responsible Body CRUD API
- What was done: Created Next.js App Router route handlers for responsible bodies: `src/app/api/responsible-bodies/route.ts` (GET all bodies with optional `?body_type=` and `?include_placeholders=` filters, includes asset counts across all 5 body roles via `_count`; POST create requiring `name` and `body_type` with uniqueness validation) and `src/app/api/responsible-bodies/[id]/route.ts` (GET single body with role-specific asset counts exposed as `asset_counts_by_role`; PUT update with placeholder deactivation guard — requires `confirm_deactivate_placeholder: true` flag to deactivate a placeholder body). Ran `tsc --noEmit` → clean.
- Files affected: `app/src/app/api/responsible-bodies/route.ts`, `app/src/app/api/responsible-bodies/[id]/route.ts`
- Learnings for next cycles:
  - Traps: Prisma enums are NOT exported from `@prisma/client` — they live in `@/generated/prisma/client` (the custom output path configured in schema.prisma). Always import enums from that path.
  - Patterns: Placeholder body guard: check `existing.isPlaceholder && body.is_active === false && body.confirm_deactivate_placeholder !== true` to block unintended deactivation.

- What was done: Created `src/lib/prisma.ts` (Prisma client singleton — cast constructor to `any` to bypass Prisma 7 TS type requiring `accelerateUrl`) and `src/lib/api-response.ts` (shared `ok()`/`err()` helpers producing `{ data, error, meta }` envelope). - Learnings for next cycles:
  - Patterns: Next.js 15 App Router dynamic route params are now async — `params` must be `await`-ed in route handlers: `const { id } = await params`.
  - Patterns: Response envelope shape: `{ data, error, meta }` — `meta` is `null` for non-paginated responses.
  - Traps: Prisma 7 PrismaClient constructor TS type requires `accelerateUrl` field. Workaround: `new (PrismaClient as any)()` cast.
